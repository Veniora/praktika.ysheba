ИНСТРУКЦИЯ ПО ЗАПУСКУ И ИСПОЛЬЗОВАНИЮ БРОКЕРА СООБЩЕНИЙ

Привет! Это документация для моего учебного проекта распределенного брокера сообщений на Go. Тут написано как все устроено, как это запускать и как это тестировать. Оформление простое, без таблиц и прочих наворотов, просто текст и код как есть.

КАК УСТРОЕН ПРОЕКТ И КЛЮЧЕВЫЕ КОМПОНЕНТЫ

Я разделил код на три папки и пару пакетов, как просили в задании:
1. Менеджер Очередей (папка cmd/manager). Это типа главный контроллер. Он слушает порт 8080. В нем крутятся глобальные карты для хранения информации о том, какие брокеры активны, какие топики к ним привязаны и какие офсеты прочитала каждая группа. Еще он выдает аренду на сообщения (leases) чтобы подписчики из одной группы не читали одно и то же сообщение. Состояние сохраняется прямо в JSON файл (data/manager/state.json) каждый раз когда кто-то присылает подтверждение (ACK).
2. Сервис-Брокер (папка cmd/broker). Это узел который хранит сообщения. Принимает запросы по HTTP на ручку /publish и пишет их в текстовый файл топика. Для простоты сообщения пишутся в формате "офсет|сообщение" на новой строке. Когда подписчик хочет прочитать сообщения через /fetch, брокер идет к менеджеру очередей, берет свободные офсеты, считывает их из файла топика (просто пробегая весь файл построчно, что очень просто и надежно) и отдает назад.
3. Библиотека SDK (папка pkg/sdk). Это просто обертка на HTTP запросах. Чтобы издателям и подписчикам было проще жить. Подписчик внутри запускает бесконечный цикл, который опрашивает брокер, передает сообщения в функцию-обработчик и если она не упала, отправляет ACK назад на брокер (который пересылает его в менеджер).

ГДЕ ХРАНЯТСЯ ДАННЫЕ
Все файлы баз данных запрещены, поэтому все сохраняется в файлы на диске:
* Смещение групп подписчиков пишется прямо в json-файл data/manager/state.json.
* Сами логи топиков лежат в папке data/broker/<ID-брокера>/<имя_топика>.txt. Каждое сообщение — это просто строчка.

СПИСОК РУЧЕК HTTP И АПИ

Менеджер Очередей (порт 8080 по умолчанию):
* /brokers/register (POST) - брокер регистрируется при старте
* /brokers/heartbeat (POST) - брокер каждые 2 секунды шлет пинг
* /topics/register (POST) - привязка топика к брокеру
* /topics/route?topic=xxx (GET) - узнать на каком брокере лежит топик. Если брокер упал или топика нет, менеджер очередей сам выберет случайный рабочий брокер и привяжет топик к нему!
* /qm/lease (POST) - брокер берет офсеты в аренду для подписчика
* /qm/ack (POST) - брокер подтверждает обработку сообщений и сдвигает офсет
* /status (GET) - статистика системы (активные брокеры, топики, группы, их офсеты)

Брокер (порты 8081, 8082 и т.д.):
* /publish (POST) - опубликовать сообщение (издатель шлет сюда)
* /fetch (POST) - прочитать сообщения (подписчик шлет сюда)
* /ack (POST) - подтвердить обработку (подписчик шлет сюда)

БЫСТРЫЙ И ИНТЕРАКТИВНЫЙ ЗАПУСК (ДЛЯ ЛЮБОГО КОМПЬЮТЕРА)

Я добавил специальный скрипт quick-start.sh, с помощью которого любой человек без установленного Go (просто с установленным Докером) может запустить все за 5 секунд.

Выполни в терминале:
./quick-start.sh

И выбери нужный пункт:
1) Запустить локальные тесты на хосте (если установлен Go)
2) Собрать и поднять весь кластер в Докере (Go не нужен, сборка идет внутри контейнеров)
3) Запустить демо-клиент отправки/получения в Докере (для проверки работы докер-окружения)
4) Остановить контейнеры и очистить все временные папки
5) Выйти

КАК БЫСТРО ЗАПУСТИТЬ ВСЕ ТЕСТЫ НА ОДНОЙ МАШИНЕ (ЛОКАЛЬНО)

Для этого я написал скрипт test.sh. Он делает все сам: собирает бинарники, чистит старые данные, запускает менеджер и брокеры, прогоняет тесты и останавливает процессы в конце.

Нужно просто дать ему права на выполнение и запустить:
chmod +x test.sh
./test.sh

После запуска вы увидите в консоли отчет с зелеными надписями [SUCCESS] по всем 4 сценариям проверки:
1. Тест сохранения данных (Data Safety) - отправляем 1000 сообщений, убиваем брокер с концами через kill -9, запускаем заново, подписчик читает все 1000 сообщений без ошибок.
2. Балансировка нагрузки (Load Balancing) - запускаем 2 копии подписчиков в одной группе с лимитом 1. Отправляем 10 сообщений. Каждый обрабатывает ровно по 5 сообщений, потому что менеджер очередей выдает офсеты строго по очереди.
3. Продолжение с места остановки (Restart from Offset) - читаем 5 сообщений подписчиком, гасим его, отправляем новые сообщения, запускаем снова и он начинает читать с 6 сообщения, а не сначала.
4. Разные группы (Multiple Groups) - одна тема читается подписчиками из группы А и группы Б независимо друг от друга.

КАК ЗАПУСТИТЬ ЧЕРЕЗ DOCKER COMPOSE И ТЕСТИРОВАТЬ ВНУТРИ СЕТИ

Если хочется запустить в контейнерах, то я написал docker-compose.yml. Запуск простой:
docker compose up --build -d

Эта команда поднимет три контейнера:
1. queue-manager - менеджер очередей (порт 8080)
2. broker-1 - первый брокер (порт 8081)
3. broker-2 - второй брокер (порт 8082)

Брокеры зарегистрируются в менеджере и будут доступны для работы. Данные сохраняются на хост-машину в созданную папку data/.

Вы можете отправлять и читать сообщения с помощью скрипта-помощника docker-demo.sh. Он прокидывает команды во внутреннюю сеть Докера, так что вам не нужен Go на хосте.

Примеры использования:
* Опубликовать 5 сообщений:
./docker-demo.sh -mode publish -topic test-topic -count 5 -payload "msg-data"

* Прочитать эти сообщения:
./docker-demo.sh -mode subscribe -topic test-topic -group my-group -id my-sub -count 5

* Посмотреть статус кластера:
./docker-demo.sh -mode status

КАК РАЗВЕРНУТЬ НА РАЗНЫХ ВИРТУАЛКАХ (VM) ИЛИ НА РАЗНЫХ ПОРТАХ

Если нужно запустить руками на разных машинах (или на разных портах одной машины):

Шаг 1. Собираем бинарники:
go build -o manager cmd/manager/main.go
go build -o broker cmd/broker/main.go
go build -o demo cmd/demo/main.go

Шаг 2. Запускаем Менеджер Очередей на машине 1 (IP: 192.168.1.100, порт 8080):
./manager -port 8080 -state data/manager/state.json

Шаг 3. Запускаем первый Брокер на машине 2 (IP: 192.168.1.101, порт 8081):
./broker -id broker-1 -port 8081 -qm http://192.168.1.100:8080 -addr http://192.168.1.101:8081 -data data/broker

Шаг 4. Запускаем второй Брокер на машине 3 (IP: 192.168.1.102, порт 8082):
./broker -id broker-2 -port 8082 -qm http://192.168.1.100:8080 -addr http://192.168.1.102:8082 -data data/broker

Шаг 5. Используем демонстрационную программу для отправки и чтения:

* Чтобы опубликовать 10 сообщений в топик "my-topic" через Менеджер Очередей:
./demo -mode publish -topic my-topic -count 10 -payload "test-msg" -qm http://192.168.1.100:8080

* Чтобы запустить подписчика в группе "group-xyz" с именем "sub-client-1":
./demo -mode subscribe -topic my-topic -group group-xyz -id sub-client-1 -count 10 -qm http://192.168.1.100:8080

* Чтобы посмотреть текущее состояние очередей и смещения:
./demo -mode status -qm http://192.168.1.100:8080

НЕБОЛЬШИЕ ОШИБКИ И СЛАБЫЕ МЕСТА КОДА (ОТ ДЖУНА)

Код рабочий и проходит все тесты, но в нем есть косяки, которые опытный прогер бы переделал:
1. Чтение лог-файла брокера целиком при каждом запросе (/publish и /fetch). Если сообщений будет миллион, то чтение файла построчно с самого начала будет тормозить брокер.
2. Отсутствие атомарной записи JSON-файла с офсетами в менеджере. Метод ioutil.WriteFile пишет сразу в боевой файл. Если в этот момент сервер вырубить по питанию, файл offsets.json может остаться пустым или сломанным.
3. Глобальные переменные и одна большая блокировка GlobalLock в Менеджере и Mutex в Брокере. Это тормозит конкурентные запросы.
4. В SDK при ошибке запросов к брокеру мы просто спим 200 миллисекунд и пробуем снова. Настоящий SDK должен уметь экспоненциальный бэкафф.
5. В demo/main.go при выходе подписчика стоит "time.Sleep(200 * time.Millisecond)", чтобы фоновая горутина успела дослать ACK. Это костыль, правильнее использовать wait groups или каналы для завершения.
